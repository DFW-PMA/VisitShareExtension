## 2. MANDATORY CODING CONVENTIONS

These apply to **every file touched**, no exceptions.

### 2a. Hungarian Prefixes
All variable/property names use Hungarian prefixes:
- `s` — String
- `b` — Bool
- `i` — Int
- `c` — count (Int used as count)
- `dbl` — Double
- `arr` — Array
- `dict` — Dictionary
- `obj` — generic object reference

### 2b. ClassInfo — Legacy Pattern (grandfathered) and New Macro Pattern

#### Legacy pattern (hand-typed — grandfathered in existing files)
Older files contain a hand-typed `struct ClassInfo` block directly inside every class or struct:

```swift
struct ClassInfo
{
    static let sClsId        = "ClassName"
    static let sClsVers      = "v1.0101"    // see versioning rules below
    static let sClsDisp      = sClsId+".("+sClsVers+"):"
    static let sClsCopyRight = "Copyright (C) JustMacApps 2023-2026. All Rights Reserved."
    static let bClsTrace     = true
    static let bClsFileLog   = true
}
```

And at the top of every method body, the hand-typed two-line preamble:

```swift
let sCurrMethod:String     = #function
let sCurrMethodDisp:String = "\(ClassInfo.sClsDisp)'"+sCurrMethod+"':"
```

**If you see this pattern in a file it has NOT yet been migrated to the macro.** Do not modify it
manually — migration is performed by `JmMigrateEntityInfoCLI` (see Section 11). Do not introduce
new hand-typed `ClassInfo` blocks in any new file — use the macro pattern below instead.

#### New macro pattern (current standard — use in all new files)
The `JmEntityInfo` macro package replaces the hand-typed boilerplate. Add the import and attach
the macro to the enclosing type declaration. The `struct ClassInfo` block is generated automatically
at compile time and remains accessible at all existing call sites (`ClassInfo.sClsVers`, etc.)
without any changes to those call sites.

```swift
import JmEntityInfo

@JmEntityInfo(vers:"v1.0101")
struct MyNewView: View
{
    // No hand-typed struct ClassInfo needed here — generated by the macro.
    ...
}
```

At the top of every method body, replace the two-line preamble with the expression macro:

```swift
let sCurrMethodDisp:String = #JmCurrentMethodInfo
```

This expands at compile time to the same string the old two-line preamble produced.
`#function` resolution is per-call-site, so every method gets its own correct name automatically.

#### Call-site compatibility
All existing call sites (`ClassInfo.sClsId`, `ClassInfo.sClsVers`, `ClassInfo.sClsDisp`,
`ClassInfo.sClsCopyRight`, `ClassInfo.bClsTrace`, `ClassInfo.bClsFileLog`) are **unchanged** —
the macro generates the same nested `struct ClassInfo` the hand-typed code did. No call site
edits are required after migration.

#### Macro package location
- GitHub repo: `JmEntityInfo` (standalone Swift macro package)
- Add to `Package.swift` dependencies as needed per project
- Macro symbol: `@JmEntityInfo` (member macro, attaches to struct/class)
- Expression macro: `#JmCurrentMethodInfo` (use inside any method body)

### 2c. Version String Format — `v#.iijj`
- `#` — major version (increment on significant redesign)
- `ii` — day-touch counter: increments by 1 each **distinct calendar day** the file is modified (not a date, just a count)
- `jj` — intra-day edit counter: resets to `01` on each new day touched; increments for each additional edit within that same day

Examples:
- `v1.0101` — touched on 1 day, first edit of that day
- `v1.0102` — touched on 1 day, second edit same day
- `v1.0201` — touched on 2 different days, first edit today
- `v1.0302` — touched on 3 different days, second edit today

**Always bump the version string when modifying a file.**

### 2d. Logging — appLogMsg() ONLY
- Use `appLogMsg()` (via XCGLogger) exclusively — no `print()`, no `NSLog()`, no `debugPrint()`
- Log format: `"\(ClassInfo.sClsDisp)'\(#function)': message"` (or via `sCurrMethodDisp`)

### 2h. Singleton ObservableObject Pattern
Managers are singletons conforming to `ObservableObject`:

```swift
class AppSomeManager: ObservableObject
{
    static let sharedInstance = AppSomeManager()
    private init() { }
    // ...
}
```

### 2i. Singleton Access Pattern
Newer singletons expose their instance as a `static let` named with the class name lowercased first letter, directly on the class:

```swift
static let appSomeManager:AppSomeManager = AppSomeManager()
// accessed as: AppSomeManager.appSomeManager
```

Very old code uses a nested `ClassSingleton` struct wrapper — that pattern is being phased out. **Never assume `ClassSingleton` exists.** Always grep for `static let` in the actual file before referencing the singleton. Do not introduce `ClassSingleton` in any new code.

## 3. ACTIVE REFACTOR — AppPersistentDataManager
> **NOTE:** This section describes work originating in VMA (VisitManagementApp / DFW-PMA healthcare
> suite). The `AppPersistentDataManager`, `PersistentDataManager` protocol, `AppJsonDataManager`,
> and related files are **shared common infrastructure** that also applies to PACK Apps wherever
> local persistence is needed. Read this section for architecture context even when working on a
> PACK App. Do not assume these files don't exist in a PACK App — grep first.

### 3a. Context
VMA currently uses `AppSwiftDataManager` directly in approximately **24 source files**. The goal is to introduce a persistence abstraction layer so backends are pluggable and SwiftData can be phased out in favor of JSON on disk.
However, `JmAppSwiftDataManager` is a different and older module. Do not touch any references to `JmAppSwiftDataManager` as it will eventually be deleted.

This work is **in progress but not yet started in code**. Architecture has been agreed; implementation begins now.

### 3b. Target Stack

```
AppDataItemsRepo                  ← in-memory working surface, UNCHANGED
        ↓
AppPersistentDataManager          ← NEW coordinating facade (singleton ObservableObject)
        ↓  (PersistentDataManager protocol)
AppSwiftDataManager               ← existing backend, untouched initially
AppJsonDataManager                ← NEW JSON-on-disk backend, target end state
```

### 3c. New Protocol: `PersistentDataManager`

A clean new protocol (not a rename). `AppSwiftDataManager` will be adapted to conform. `AppJsonDataManager` will implement it from scratch. Callers above (`AppDataItemsRepo`) will reference `AppPersistentDataManager` only — they never touch the backends directly.

Minimum protocol surface (to be confirmed before coding):
```swift
protocol PersistentDataManager
{
    func load<T: Codable>(_ type: T.Type) async throws -> [T]
    func save<T: Codable>(_ items: [T]) async throws
    func upsert<T: Codable>(_ item: T) async throws
    func delete<T: Codable>(_ item: T) async throws
}
```

### 3d. PersistenceBackend Enum

```swift
enum PersistenceBackend
{
    case swiftData
    case json
    // <<STUB v1>> Future backends added here as comments until needed
    // case coreData  — explicitly ruled out (old tech)
}
```

### 3e. AppPersistentDataModeler
Replaces `AppSwiftDataModeler` as the per-type registry. Marked `<<< App 'dependent' >>>`. Contains a `[String: PersistenceBackend]` config map — one line per type to switch its backend. No other code changes required to flip a type.

### 3f. SwiftData Inventory in VMA
These are the SwiftData-backed types that HAVE been migrated:

| Type | Notes |
|---|---|

### 3g. Migration Rules
- Do **not** rip out SwiftData while it is still active — swap per type via the config map
- Do **not** add SwiftData to anything new
- `AppSwiftDataManager` stays intact (commented out progressively as types migrate)
- CoreData is **not** a future backend option — ruled out

### 3h. DivorcePack as Reference Pattern
DivorcePack is the proven target architecture: no SwiftData, JSON on disk, in-memory repo as working surface. When in doubt about how the JSON layer should behave, DivorcePack's implementation is the reference.

## 11. JmEntityInfo MACRO MIGRATION

### 11a. Context

VV and VMA are deliberately held back until the CLI has earned further trust on PACK Apps.

### 11b. What the Migration Does
For every `struct` or `class` in a file that directly contains a `struct ClassInfo`:

1. Inserts `import JmEntityInfo` as the first import line in the file (skipped if already present)
2. Prepends `@JmEntityInfo(vers:"<extracted sClsVers>")` immediately before the type declaration
3. Comments out (default) or deletes every line of the `struct ClassInfo { }` block
4. In every method body of that type, comments out (default) or deletes the two-line preamble:
   ```swift
   let sCurrMethod:String     = #function
   let sCurrMethodDisp:String = "\(ClassInfo.sClsDisp)'"+sCurrMethod+"':"
   ```
   and inserts immediately after:
   ```swift
   let sCurrMethodDisp:String = #JmCurrentMethodInfo
   ```

Types without a direct `struct ClassInfo` child are left untouched.

### 11c. JmMigrateEntityInfoCLI — Usage

**Dry-run first (default — no files written):**
```bash
jm-migrate-entity-info \
    --source-dir /path/to/AppName/Sources \
    --backup-dir /path/to/Backups/AppName \
    --pattern "*.swift" \
    --verbose
```

**Apply (writes changes + takes backups):**
```bash
jm-migrate-entity-info \
    --source-dir /path/to/AppName/Sources \
    --backup-dir /path/to/Backups/AppName \
    --pattern "*.swift" \
    --apply
```

**With recursion (subdirectories):**
```bash
jm-migrate-entity-info \
    --source-dir /path/to/AppName \
    --backup-dir /path/to/Backups/AppName \
    --pattern "*.swift" \
    --recursive \
    --apply
```

**Delete originals instead of commenting out:**
```bash
    --delete-originals   # add to any of the above
```

**Backup collision scheme:** `Foo.swift` → `Foo_0.swift`; if `Foo_0.swift` already exists →
`Foo_1.swift`, etc. The tool is idempotent — re-running on already-migrated files is safe;
files with no qualifying `struct ClassInfo` are reported as `[SKIP]` and left untouched.

## 12. SWIFT 6 MIGRATION — INVENTORY AND ASSESSMENT

### 12a. Purpose
This section governs a **discovery-only** pass over the PACK App source files to assess what
Swift 6 strict concurrency migration would require. **No fixes are made during this pass.**
The output is a per-app inventory report that will be reviewed before any migration work is
committed to.

### 12b. Background
The PACK Apps are currently built with Swift 5 language mode. Swift 6 introduces strict
concurrency checking — every potential data race becomes a compile-time error rather than a
runtime hazard. The impact on a given app depends heavily on how much concurrent code it
contains and whether its types cross isolation boundaries.

Key Swift 6 concurrency concepts Claude Code must understand before inventorying:
- **`Sendable` conformance** — types crossing actor/concurrency boundaries must conform to
  `Sendable` (or be `@unchecked Sendable` as a temporary measure)
- **Actor isolation** — `@MainActor`, custom actors, isolation inference
- **`async`/`await` and `Task`** — existing `DispatchQueue.main.async` patterns may need
  revisiting under strict checking
- **`nonisolated`** — methods that can safely run outside an actor's isolation
- **`@preconcurrency`** — import modifier to suppress warnings from pre-concurrency frameworks

### 12c. Inventory Procedure
For each PACK App in the migration order below:

1. **Enable Swift 6 language mode temporarily** (do NOT commit this change):
   In `Package.swift` or Xcode project settings, set Swift language version to 6.
   Build. Collect all compiler errors and warnings. Do not fix anything.

2. **Categorize each diagnostic** into one of:
   - `SENDABLE` — type needs `Sendable` conformance
   - `ACTOR-ISOLATION` — call to main-actor-isolated code from non-isolated context (or vice versa)
   - `ASYNC-CONTEXT` — `async` function called without `await`, or `await` in non-async context
   - `DISPATCH-REPLACE` — `DispatchQueue.main.async` that should become `Task { @MainActor in ... }`
   - `PRECONCURRENCY` — framework import needs `@preconcurrency`
   - `OTHER` — anything that doesn't fit the above

3. **Count and list** — for each category, list the affected files and a representative
   diagnostic message. A simple count per category is sufficient; do not attempt exhaustive
   line-by-line listing unless the count is under 10.

4. **Revert** the Swift 6 language mode change before ending the session.

5. **Produce a per-app summary** in the format below and paste it into Section 12d.

### 12g. Conversion Fix Conventions — Minimal-Whitespace Closure-Type Syntax
Once an app passes the decision gate and conversion (real fixes) begins, one parser-strictness
pattern is expected to recur across many files, not just `JmAppSyncUpdatesOnMainThread.swift`:

**The issue:** `@MainActor()->Void` (and similarly `@escaping`, `@Sendable`, etc. directly butting
against the parameter-list's opening paren with no space) parsed leniently under Swift 5 but is a
hard parse error under Swift 6 (`error: expected parameter type following ':'`). Because it's a
parse error, it blocks type-checking of the **entire file** (sometimes the whole module, depending
on how Xcode batches compile jobs) — no other Swift 6 diagnostic in that file can surface until
this is fixed first.

**The fix — ONE mandatory space only:** Add a single space between the attribute and the opening
paren of the parameter list: `@MainActor ()->Void`. Daryl's house style is otherwise tight/minimal
whitespace (no spaces around `->`, no space before `(`) — Swift 6 does not require those, only the
one space after the attribute. **Do not add whitespace anywhere else** when fixing this pattern;
match the existing tight style for everything except the one space Swift 6 actually demands.

```swift
// Before (Swift 5 — parses, but breaks under Swift 6):
func foo(_ operation:@MainActor()->Void)

// After (Swift 6 — minimal fix, only the mandatory space added):
func foo(_ operation:@MainActor ()->Void)

// NOT this (over-fixed — adds unnecessary whitespace around the arrow):
func foo(_ operation:@MainActor () -> Void)
```

**When converting any app:** grep for the pattern `@MainActor(`, `@Sendable(`, `@escaping(` (attribute
immediately followed by `(` with no space) across the source tree before starting fixes — this lets
the parse-blocker sites be found and fixed in one pass up front, rather than discovered one at a time
as each one blocks the next diagnostic from appearing.

## 13. AppPersistentDataManager — GENERIC JSON DISPATCH REFACTOR

### 13a. Problem (agreed 2026-06-25)
`AppPersistentDataManager.swift` and `AppJsonDataManager.swift` (both in `Models/`) are meant to be
shared, syncable files identical across every App (per Section 3). `AppPersistentDataModeler.swift`
(in `AppCustomize/`) is deliberately the *one* per-App file, describing via
`dictPersistenceBackend`/`getSwiftDataModelTypes()`/`getJsonModelTypeNames()` which item types exist
in this App and whether each is `.json` or `.swiftData`. That list legitimately varies per App — a
JSON-backed type might not exist at all in a given App.

The bug: `AppPersistentDataManager.swift`'s delegation methods (`fetch`, `upsert`, `delete`,
`upsertPersistentModel`, `retrieveStorage`) dispatch JSON-backed types via a `switch` on the type-name
string, with each `case` doing `item as? ConcreteTypeName` (e.g. `as? CLRequestGoodItem`). That cast
requires the concrete type to actually exist and compile in whatever App the file sits in — so a
"shared" file silently grows a hard compile dependency on every JSON type from every App, exactly the
thing the per-App `AppPersistentDataModeler` design was meant to avoid.

### 13b. Fix (agreed, not yet implemented)
Replace the string-switch + named-type cast with existential opening — no concrete type names appear
in `AppPersistentDataManager.swift` or `AppJsonDataManager.swift` ever again:
- Instance-based calls (`upsert`, `delete`): cast the value, `item as? any JsonDataItem`.
- Type-based calls (`fetch`, `retrieveStorage`, which only have `T.self`, no instance): cast the
  metatype, `T.self as? any JsonDataItem.Type`.
- Add type-erased helpers to `AppJsonDataManager` (`loadAny`, `saveAny`, `upsertAny`, `deleteAny`)
  that open the existential into the existing generic `load<T:JsonDataItem>`/`save`/`upsert`/`delete`.
- `AppPersistentDataModeler.swift` is unchanged in shape — it remains the only per-App file.

Net effect: `AppPersistentDataManager.swift`/`AppJsonDataManager.swift` become byte-identical,
JMACodeSync-able across every App with zero post-sync edits. Adding a JSON type to an App becomes:
(1) make it `Codable + JsonDataItem`, (2) flip one entry in that App's `AppPersistentDataModeler` map.

## 14. macOS VIEW PRESENTATION — `.sheet()` vs `openWindow()`

### 14a. The Problem (found 2026-06-25, JustAMultiplatformClock1)
A non-trivial view (`AppDocumentsFileReaderView` — `NavigationStack` + `List` with dynamic content)
presented via `.sheet(isPresented:)` on macOS rendered almost empty — a header showed an item count
(e.g. "8 items") but the `List` itself had no usable space and nothing else appeared. Presenting the
exact same view via a separate `Window` scene + `openWindow(id:)` instead rendered everything
correctly.

### 14b. Root Cause
This is a known macOS SwiftUI sizing quirk, not a `.sheet()`/`openWindow()` reliability difference:
macOS sheets size themselves to the *ideal size* their content reports. A `NavigationStack`+`List`
with dynamic/data-driven content frequently can't report a sensible ideal size on macOS the way it
does on iOS (where `.sheet()` gets full-screen-card UIKit presentation behavior instead), so the
sheet collapses to a tiny default frame. A separate `Window` scene gets real AppKit window sizing
and resizing, so the same view content renders fine there.

### 14c. The Convention
- **Prefer `openWindow(id:)` + a `Window(...)` scene over `.sheet()` for any non-trivial view
  presented on macOS** — anything with a `List`, `NavigationStack`, dynamic/data-driven content, or
  generally more than a simple fixed-layout form/alert.
- **`.sheet()` is fine on macOS** for simple, small, fixed-size content (confirmation dialogs,
  short forms, About panels) where the ideal size is unambiguous.
- If `.sheet()` must be used on macOS for non-trivial content anyway, give the content an explicit
  `.frame(minWidth:..., minHeight:...)` — this is the direct fix for the sizing problem, but
  `openWindow()` sidesteps it entirely and is the more robust default.
- This is a platform-conditional decision — iOS `.sheet()` does not have this problem and should
  keep using `.sheet()` as normal; only the macOS path needs the `openWindow()`/`Window` treatment.
- Pattern to use:
  ```swift
  // Button action:
  #if os(macOS)
      openWindow(id:"AppDocumentsFileReaderView")
  #elseif os(iOS)
      isDocumentsFileReaderShowing = true   // still .sheet() on iOS
  #endif

  // In the App's Scene body (macOS-only Window, alongside the existing WindowGroup):
  #if os(macOS)
  Window("AppDocumentsFileReader", id:"AppDocumentsFileReaderView")
  {
      AppDocumentsFileReaderView()
          .environment(\.appGlobalDeviceType, appGlobalDeviceType)
  }
  #endif
  ```
- **Check for this when converting/auditing any PACK App**: grep for `.sheet(` across the source
  tree and cross-reference each presented view's body for `List`/`NavigationStack`/dynamic content
  — any hit is a candidate for the `openWindow()` treatment on macOS.

## 13. SWIFT 6 BUILD SETTINGS — Diagnosing Single-App Compile Errors

### 13a. The Symptom
A shared/synced source file (via JMACodeSync) compiles and runs fine in a dozen other PACK Apps /
DFW-PTS apps for days, then suddenly throws Swift 6 actor-isolation errors ("Call to main
actor-isolated ... in a synchronous nonisolated context") in exactly one app, with no recent edits
to that file.

### 13b. The Rule
**Before touching any code**, check that app's `.xcodeproj/project.pbxproj` for `SWIFT_*` build
settings that diverge from the other apps — especially:
- `SWIFT_DEFAULT_ACTOR_ISOLATION` (e.g. `MainActor`) — makes otherwise-`nonisolated` functions
  implicitly main-actor-isolated in that target ONLY
- `SWIFT_VERSION`
- `SWIFT_STRICT_CONCURRENCY`
- `SWIFT_UPCOMING_FEATURE_*`
- `SWIFT_APPROACHABLE_CONCURRENCY`

Diff the failing app's target build settings against a known-working sibling app's settings. If
they diverge, that is almost certainly the root cause — not a real bug in the shared source file.

### 13c. Confirmed Incident (2026-06-26)
JMACodeSync's app target alone had `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` set in its
Debug/Release `XCBuildConfiguration` blocks. No other PACK App / DFW-PTS app had it. This made
`appLogMsg()` and `JmAppDeepCopyUtility.deepCopy()` implicitly `@MainActor`-isolated only in
JMACodeSync, breaking a `DispatchQueue.global().async { }` call site in
`JmAppDeepCopyExamples.swift` that was fine everywhere else. **Fix:** removed the setting from
`project.pbxproj` directly — zero source code changes. Confirmed working immediately.

### 13d. Session Startup Addendum
When a compile error appears that looks like a Swift 6 concurrency violation, and the same file is
known to work elsewhere: grep `project.pbxproj` for the `SWIFT_*` settings above and compare against
a sibling app before proposing any code change.
